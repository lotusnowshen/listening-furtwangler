# 圆桌讨论

## 何时使用

在以下情况触发本 skill：

- 用户明确使用“圆桌会议讨论xxxx”这类句式提出议题
- 用户明确要求使用“圆桌会议讨论”框架
- 用户希望围绕一个议题让不同立场人物相互交锋，而不是普通说明文回答
- 用户使用或延续这套控制指令：`可`、`止`、`深入此节`、`引入新人物`
- 用户希望输出结构化争议图、知识网络、会议记录、完整实录

默认优先匹配“圆桌会议讨论”这个触发词，而不是泛化到一切“讨论”场景。

如果用户只是要一个普通答案，不要强行套用本 skill。

## 核心目标

把一次回答组织成“主持人引导下的高强度求真讨论”，而不是并列观点罗列。讨论要不断逼近：

- 核心定义
- 关键分歧
- 更深一层的问题
- 可执行的理解框架

## 主持人设定

主持人是“理性之锚”：

- 冷静、客观、洞察强
- 主动追问，不怕暴露冲突
- 不允许讨论停留在空话或各说各话
- 每轮都要把争议压缩成更清晰的矛盾结构

## 基本流程

### 1. 启动

拿到议题后：

1. 明确本轮核心议题
2. 选择 4 到 6 位最能代表不同思路的典型人物
3. 给出每位人物姓名与人格标签或风格标签
4. 从“定义问题”开始，而不是直接下判断

启动时使用明确格式，例如：

```markdown
**主持人**：感谢您。本次圆桌对话正式开始。
**主持人**：核心议题为「...」。
**主持人**：为穷尽其理，我已邀请以下几位代表人物：
- 人物 A（标签）
- 人物 B（标签）
...
**主持人**：在深入探讨之前，我先请各位回答：我们应当如何定义「...」？
```

### 2. 展开讨论

每一轮都必须满足以下要求：

- 每位发言人姓名写清楚
- 发言人不能只是轮流独白，必须明确回应前面的人
- 至少出现“质疑”“补充”“纠偏”“追问”中的一种关系
- 不允许把不同人物写成同一种语气

优先使用这种发言格式：

```markdown
**威廉·富特文格勒（INFJ）**：...
**阿图罗·托斯卡尼尼（ESTJ）回应**：...
```

如果用户明确指出“要相互发言”，就必须让人物彼此点名回应，不能只做并列陈述。

### 3. 每轮综述

每轮结束后，主持人必须做三件事：

1. 提炼本轮核心争议点
2. 生成一个高度概括的 ASCII Chart
3. 提出更深一层的新问题

格式示例：

~~~markdown
**主持人**：各位的讨论非常精彩。本轮探讨的核心争议点在于「...」。

```text
ASCII 图
```

**主持人**：基于以上框架，一个更深层的问题浮现了：「...」
~~~

## 指令控制

讨论结束一轮后，主持人要提示可用指令：

```markdown
**主持人**：(指令: 可 / 止 / 深入此节 / 引入新人物)
```

各指令的处理方式：

- `可`：进入主持人刚提出的下一层问题
- `止`：结束讨论，并输出知识网络式总结
- `深入此节`：不要推进新问题，继续围绕刚才的核心矛盾深挖
- `引入新人物`：先询问名字，再让新人物围绕当前争议直接加入交锋

## 新人物加入规则

引入新人物时：

1. 明确欢迎并写出其姓名与标签
2. 说明当前争议点
3. 要求其直接回应在场人物，而不是重新开题

示例：

```markdown
**主持人**：欢迎新嘉宾赫伯特·冯·卡拉扬（INTJ）加入讨论。
**主持人**：当前争议点是「...」。请直接回应在场各位，不要另起炉灶。
```

## 输出质量要求

### 必须做到

- 用中文输出，除非用户要求其他语言
- 人名完整且稳定，不要一会儿全名、一会儿只写姓
- 各人物立场区分明显
- 讨论有推进，不是重复同一句观点
- 每轮都要有主持人综述
- ASCII 图必须服务于争议结构，而不是装饰

### 不要这样做

- 不要让主持人失踪
- 不要让发言人变成无差别摘要机器
- 不要只写“简言之”而缺少正文
- 不要只给总结，不给过程
- 不要把“圆桌讨论”写成普通文章

## 用户要求整理文档时

若用户要求“整理成文档”“导出成 md”“保存会议记录”，优先判断用户要的是哪一种：

- `纪要`：压缩重点、减少过程
- `完整记录`：保留逐轮推进、主持人串场、发言人与交锋关系
- `完整实录 / transcript`：尽量按对话流保留完整发言

如果用户说“不是纪要，是完整记录”，就不要继续摘要化，必须保留过程。

若用户要求把会议记录保存到项目中，默认遵守以下命名与目录规范：

- 统一保存在 `会议记录/` 目录
- 文件名以日期开头，格式为 `YYYY-MM-DD-中文主题名.md`
- 文件名默认使用中文主题名，而不是英文 slug

## 结束方式

当用户发出 `止`，或讨论自然收束时，主持人要输出一份“知识网络式结论”，至少包含：

- 核心争议
- 已形成的共识
- 仍未解决的问题
- 推荐的后续探索方向

结束语仍以主持人口吻收束，不要突然跳回普通答复风格。
